1. EC2 Graviton Processors

Updated SA Pro
Why Graviton matters: AWS-designed Arm-based processors delivering 20–40% better price-performance than comparable x86 instances. Three generations have launched since 2021, with Graviton5 arriving in 2026.

Graviton Generation Timeline

May 2022 — Graviton3 (GA)

25% better compute performance vs Graviton2. 2x faster floating-point and cryptographic workloads. Instance families: C7g, M7g, R7g.

July 2024 — Graviton4 (R8g GA)

96 cores, Arm Neoverse V2 architecture. 30% better compute vs Graviton3. 40% faster database workloads. 45% faster Java applications. Instance families: R8g, M8g, C8g.

June 2026 — Graviton5 (M9g GA)

192 cores, 3nm process, 3.3 GHz clock speed, PCIe Gen 6 support. Introduces the Nitro Isolation Engine with formal verification for workload isolation.

Instance Family Mapping

Generation Compute (C) General (M) Memory (R) Key Improvement
Graviton3 C7g M7g R7g 2x floating-point/crypto
Graviton4 C8g M8g R8g 40% faster DB, 45% faster Java
Graviton5 — M9g — 192 cores, 3nm, Nitro Isolation Engine
EXAM TIP: Graviton instances deliver 20–40% better price-performance than x86. When a question asks about cost optimization for compute-heavy workloads, Graviton is almost always the answer. Remember the g suffix in instance family names indicates Graviton (e.g., M7g, R8g).

2. Auto Scaling Updates

Updated SA Pro

Predictive Scaling Enhancements

  • Backfill forecasts (May 2022): Predictive Scaling can now backfill forecast data, enabling accurate predictions even when first enabled on an existing Auto Scaling group
  • More frequent forecasting (Jan 2023): Forecast intervals shortened for more responsive capacity planning

Warm Pools

What are Warm Pools? Pre-initialized EC2 instances kept in a stopped or hibernated state, ready to join the Auto Scaling group on demand. Eliminates cold start delays for applications with long initialization times.
  • Hibernation support (Feb 2022): Warm pool instances can hibernate, preserving in-memory state for even faster launch
  • Return to pool on scale-in: Instead of terminating instances during scale-in, return them to the warm pool for reuse
EXAM TIP: Classic exam pattern: "Application takes 10 minutes to initialize. How do you reduce scaling time?" The answer is warm pools. Instances are pre-initialized (stopped or hibernated) and can join the group in seconds rather than minutes.

4. S3 Updates

Major SA Pro

S3 Express One Zone (GA November 2023)

Game-changer: A new storage class purpose-built for performance-critical applications requiring single-digit millisecond latency.
Feature Detail
Latency Single-digit milliseconds — 10x faster than S3 Standard
Request cost 50% lower request costs vs Standard
Throughput Up to 2 million requests/sec
Bucket type Directory buckets (not general-purpose buckets)
Availability Single-AZ (trade durability for performance)
Authentication Session-based auth (CreateSession API)
Use cases ML training data, financial analytics, media processing, real-time analytics

S3 Object Lambda (GA March 2021)

  • Attach Lambda functions inline with S3 GET, HEAD, and LIST requests
  • Transform data on retrieval without creating duplicate copies
  • Use cases: PII redaction, format conversion (XML to JSON), image resizing, data enrichment

S3 Glacier Instant Retrieval (November 2021)

  • Archive storage class with millisecond retrieval
  • 68% cheaper than S3 Standard-IA for rarely accessed data
  • Minimum storage duration: 90 days
  • Same retrieval performance as S3 Standard and Standard-IA

Mountpoint for Amazon S3 (GA August 2023)

  • Mount S3 buckets as a local Linux filesystem using FUSE
  • Supports sequential reads and writes (not random writes)
  • Throughput up to 100 Gbps on supported instance types
  • Ideal for data lakes, ML training pipelines, and analytics workloads that use POSIX file APIs

S3 Access Grants (GA November 2023)

  • Map corporate identities (from IAM Identity Center) directly to S3 permissions
  • Grant access by user/group at prefix, bucket, or object level
  • End-user identity appears in CloudTrail logs (not just the assumed role)
  • Ideal for enterprise data lakes with many users needing fine-grained access

Security Defaults

Change Date Detail
Default encryption Jan 2023 ALL new objects automatically encrypted with SSE-S3. Cannot be disabled.
Block Public Access Apr 2023 Block Public Access enabled by default on all new buckets. ACLs disabled by default.
S3 Bucket Keys — Reduce KMS API costs by up to 99%. Enabled by default for SSE-KMS encryption.
EXAM TIP: Know all S3 storage classes and their latency/cost tradeoffs. Express One Zone = lowest latency (single-digit ms), single-AZ, directory buckets. Object Lambda = transform data on retrieval without duplication. Glacier Instant Retrieval = archive with ms access. Access Grants = enterprise data lake identity mapping. Remember that all new objects are encrypted by default (SSE-S3) and Block Public Access is ON by default.

gp3 Sep 2025 upgrade: The September 2025 update dramatically expanded gp3 capabilities: IOPS from 16K to 80K, throughput from 1K to 2K MB/s, and max size from 16 TiB to 64 TiB. After the Sep 2025 update, gp3 handles up to 80K IOPS — only choose io2 when you need >80K IOPS, 99.999% durability, or sub-ms latency.

8. Aurora Updates

Major SA Pro

Aurora Serverless v2 (GA April 2022)

Major upgrade: Aurora Serverless v2 is a complete re-architecture from v1. It supports ALL Aurora features and scales in 0.5 ACU increments with millisecond scaling response. V1 is deprecated.
Feature Aurora Serverless v1 (Deprecated) Aurora Serverless v2
Scaling granularity Doubling (2, 4, 8, 16...) 0.5 ACU increments
Scaling speed Minutes Milliseconds
Read replicas Not supported Supported
Global Database Not supported Supported
Multi-AZ Not supported Supported
Mixed deployment No Mix provisioned + serverless in same cluster

Aurora I/O-Optimized (GA May 2023)

  • Eliminates per-I/O charges — pay only for instance and storage, not individual read/write operations
  • Saves up to 40% when I/O costs exceed 25% of total Aurora spend
  • Switch to I/O-Optimized once per 30 days; switch back to Standard anytime
  • Same performance, durability, and availability as Standard

Aurora Limitless Database (GA 2024, PostgreSQL only)

  • Horizontal scaling and sharding built into Aurora — no application-level sharding required
  • Architecture: Routers (distribute queries) + Shards (store and process data)
  • Requires I/O-Optimized storage configuration
  • PostgreSQL only (MySQL not supported)
EXAM TIP: Serverless v2 = variable/unpredictable workloads (supports all Aurora features). I/O-Optimized vs Standard = cost optimization question (choose I/O-Optimized when I/O >25% of spend). Limitless = horizontal scaling for massive PostgreSQL workloads. Remember v1 is deprecated — always choose v2.

11. MemoryDB

NEW SA Pro

GA August 2021. A durable, in-memory database compatible with Redis and Valkey, designed to be used as a primary database — not just a cache.

Key Characteristics

Feature Detail
Durability Multi-AZ transaction log — data survives node failures
Read latency Microseconds
Write latency Single-digit milliseconds
Availability 99.99% SLA
Compatibility Redis and Valkey compatible APIs
Use as Primary database (not just a cache)
MemoryDB vs ElastiCache — Critical Distinction:
ElastiCache = Cache (ephemeral). Data may be lost on node failure. Use in front of another primary database (e.g., RDS, DynamoDB).
MemoryDB = Durable primary database (persistent). Data survives node failures via Multi-AZ transaction log. Can replace both the cache AND the database layer.
EXAM TIP: When a question describes needing microsecond read latency with data durability, the answer is MemoryDB. When the question only needs a cache layer, the answer is ElastiCache. MemoryDB eliminates the "cache + database" architecture by serving as both.

12. RDS Updates

Updated SA Pro

Multi-AZ DB Clusters (GA 2022)

Not the same as traditional Multi-AZ: Multi-AZ DB Clusters provide 1 writer + 2 readable standbys across 3 AZs, with faster failover. Traditional Multi-AZ has 1 writer + 1 non-readable standby.
Feature Traditional Multi-AZ Multi-AZ DB Clusters
Architecture 1 writer + 1 standby (2 AZs) 1 writer + 2 readers (3 AZs)
Standby readable? No Yes — standbys serve read traffic
Failover time 60–120 seconds <35 seconds
Storage EBS only NVMe SSD + EBS (local write caching)
Supported engines All RDS engines MySQL and PostgreSQL only
Graviton support Yes Yes (recommended)

Blue/Green Deployments (GA November 2022)

  • Create a staging copy (green environment) of your production database
  • Apply changes (engine upgrades, schema changes, parameter tuning) to the green environment
  • Switch traffic with minimal downtime using built-in switchover mechanism
  • Supported engines: Aurora MySQL/PostgreSQL, RDS MySQL/PostgreSQL/MariaDB
  • With RDS Proxy + Multi-AZ = switchover in approximately 1 second
EXAM TIP: Multi-AZ DB Clusters vs traditional Multi-AZ: readable standbys and <35 sec failover are the key differentiators. Blue/Green Deployments = safe database upgrades with minimal downtime. When a question mentions "upgrade database engine version with minimal risk," the answer is Blue/Green Deployments.

  • Managed through AWS Organizations for centralized backup policies

Logically Air-Gapped Vault (GA August 2024)

Strongest ransomware protection: Logically air-gapped vaults provide logical isolation from source accounts. The vault exists in an AWS-managed account and is shared with your accounts via AWS RAM. Even if an attacker compromises your account, they cannot access or delete backups in the air-gapped vault.
  • Logical isolation from source AWS accounts
  • Shared via AWS Resource Access Manager (RAM)
  • Provides the strongest protection against ransomware and insider threats
  • Recovery points are immutable and isolated by design
EXAM TIP: Vault Lock Compliance mode = regulatory requirements where backups cannot be deleted by anyone (WORM). Logically air-gapped vaults = ransomware defense where backups must survive a full account compromise. Know the hierarchy: Governance (soft lock) < Compliance (hard lock) < Air-gapped (isolated from account).

26. Migration Tools

Updated SA Pro

Application Migration Service (AWS MGN)

  • Primary lift-and-shift migration service since March 2022
  • Replaced both CloudEndure Migration and Server Migration Service (SMS)
  • Agent-based: install the AWS Replication Agent on source servers
  • Continuous block-level replication to AWS, then launch test/cutover instances
  • Supports physical, virtual, and cloud-to-cloud migrations

Migration Hub Refactor Spaces

CLOSING November 2025: Migration Hub Refactor Spaces is being discontinued. It was used for the strangler fig pattern — incrementally migrating application components from monolith to microservices. The replacement is AWS Transform (launched May 2025), which provides broader application modernization capabilities.
