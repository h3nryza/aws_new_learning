1. IAM Identity Center (formerly AWS SSO)

Updated SA Pro DevOps Security
Rename: AWS SSO was renamed to IAM Identity Center on July 26, 2022. Despite the name change, API namespaces (sso, identitystore) did NOT change. CLI commands and SDKs still use sso prefixes.

Instance Types

Feature Organization Instance Account Instance
Scope Entire AWS Organization Single AWS account
Best for Multi-account environments (best practice) Standalone account or isolated use cases
Limit One per Organization One per account
Recommendation Always use this if you have Organizations Only when Organization instance is not available

Permission Sets

  • Permission Sets are templates that define what access users/groups have
  • When you assign a Permission Set to a user for an account, IAM Identity Center creates an IAM role in that target account
  • The role name follows the pattern AWSReservedSSO_<PermissionSetName>_<UniqueId>
  • Permission sets can include AWS managed policies, customer managed policies, inline policies, and permission boundaries

Identity Sources

  • Built-in Identity Center directory — simple user/group management directly in IAM Identity Center
  • Active Directory — AWS Managed Microsoft AD or AD Connector for on-premises AD
  • External Identity Providers — Okta, Microsoft Entra ID (formerly Azure AD), Google Workspace, CyberArk, OneLogin, Ping Identity via SAML 2.0 and SCIM for automatic provisioning

Key New Features

  • Trusted Identity Propagation: Passes the actual end-user identity through to downstream AWS services (e.g., QuickSight → Redshift → S3). CloudTrail logs the actual user, not the assumed role. Uses OAuth 2.0 token exchange.
  • Multi-Region resilience: Replicate your Identity Center instance to additional AWS Regions for disaster recovery and lower latency
  • AWS Access Portal: The user-facing SSO portal where users sign in and see all assigned accounts and applications
EXAM TIP: Know that AWS SSO was renamed to IAM Identity Center, but API namespaces (sso, identitystore) did not change. Understand Organization vs Account instances. Permission Sets become IAM roles in target accounts. Trusted Identity Propagation enables end-user identity to flow to downstream services.

2. AWS Control Tower

Updated SA Pro DevOps Security
Terminology change: "Guardrails" have been renamed to "Controls". The terms are synonymous in AWS documentation and exams. Over 750 preconfigured controls are available.

Three Control Behaviors

Control Type Mechanism When It Acts Example
Preventive Service Control Policies (SCPs) Before action — blocks disallowed API calls Prevent disabling CloudTrail in member accounts
Detective AWS Config Rules After action — detects non-compliant resources Detect S3 buckets without encryption enabled
Proactive CloudFormation Hooks NEW Before provisioning — blocks non-compliant resources during CloudFormation deployment Block creation of an RDS instance without encryption before it is created

Three Guidance Categories

  • Mandatory: Always enforced; cannot be disabled (e.g., disallow changes to IAM roles created by Control Tower)
  • Strongly recommended: Based on AWS best practices (e.g., enable encryption for EBS volumes)
  • Elective: Optional, commonly used for enterprise-specific needs

Account Factory for Terraform (AFT)

  • Terraform-based account provisioning pipeline for Control Tower
  • GitOps model: account requests submitted via Terraform, processed through a pipeline
  • Requires a dedicated AFT management account (separate from the Organizations management account)
  • Supports account customizations via custom Terraform modules
  • Integrates with CodePipeline, CodeBuild, and Step Functions
Management account exception: Root users and administrators in the management account can bypass Control Tower controls. This is by design — but it means the management account should have tightly restricted access and minimal workloads.
EXAM TIP: Know all three control types and their underlying mechanisms. Proactive controls via CloudFormation Hooks is new and exam-critical — they prevent non-compliant resources from being provisioned in the first place, unlike detective controls which only detect after the fact. AFT is the Terraform-native account provisioning solution.

3. AWS Organizations — New Policy Types

Updated SA Pro Security

Policy Types Comparison

Feature SCPs (Existing) RCPs NEW Declarative Policies NEW
What they govern Principals (IAM users, roles) Resources (S3 buckets, SQS queues, KMS keys, etc.) Service configurations (service-level settings)
How they work Filter maximum permissions for principals in member accounts Restrict access to resources regardless of who is calling (including external principals) Enforced at the service control plane level, not the API level
Can grant permissions? No — only restrict No — only restrict No — enforce configuration state
Affect management account? No No Yes
Affect service-linked roles? No Yes (resource-based evaluation) Yes
Key benefit Limit what principals can do Limit who can access resources (including cross-account) "Set once and forget" — persists even as services add new APIs
Example Deny s3:DeleteBucket for all roles Only allow access to S3 from within the organization Enforce IMDSv2 on all EC2 instances, block public AMI sharing
RCPs complement SCPs: SCPs restrict what principals can do. RCPs restrict access to resources. Together they provide comprehensive access control. RCPs are especially powerful for preventing external access to your resources — something SCPs alone cannot do.
Declarative Policies are unique: They are enforced at the service control plane level, not the API level. This means they persist even as AWS adds new APIs or features. They can also govern service-linked roles, which SCPs and RCPs cannot fully control. Current supported services include EC2 (IMDSv2, AMI sharing, serial console), VPC (public IP assignment), and EBS (default encryption).

Other Organizations Updates

  • Delegated Administrator: Expanded to all major security services — Security Hub, GuardDuty, Detective, Macie, Inspector, IAM Access Analyzer, Firewall Manager, Config, CloudFormation StackSets, and more. Best practice: designate a dedicated security account as delegated admin.
  • Root Access Management: Centralized control of member account root credentials from the management account. Ability to remove root credentials from member accounts entirely, perform privileged root actions without signing in as root.
EXAM TIP: The distinction between SCPs, RCPs, and Declarative Policies is critical for both SA Pro and Security specialty. Remember: SCPs = principals, RCPs = resources, Declarative Policies = service configuration. Declarative Policies are the only type that affects the management account and can fully govern service-linked roles.

4. IAM Changes

Updated SA Pro Security

IAM Roles Anywhere (July 2022)

Provides temporary AWS credentials for workloads running outside of AWS using X.509 certificates. This eliminates the need for long-term access keys for on-premises, edge, or multi-cloud workloads.

Component Description
Trust Anchor Reference to a Certificate Authority (CA) — can be AWS Private CA or an external CA. Establishes trust.
Profile Maps to an IAM role + optional session policy. Defines what permissions the external workload receives.
Credential Helper aws_signing_helper tool runs on the external workload. Presents the X.509 certificate and retrieves temporary credentials.
How it works: External workload presents its X.509 certificate → IAM Roles Anywhere validates against the Trust Anchor → creates temporary STS credentials mapped to the IAM role in the Profile → workload uses standard AWS SDK calls.

IAM Access Analyzer Expansions

Analyzer Type What It Finds Scope Engine
External Access Resources shared with external principals (other accounts, public access) Regional Zelkova (automated reasoning)
Internal Access Cross-account access within your organization or account Regional Zelkova
Unused Access (Nov 2023) Unused roles, permissions, passwords, and access keys NOT regional (org-wide) Usage analysis

Custom Policy Checks (Nov 2023)

Note: Custom Policy Checks is a paid feature priced per API call. It uses automated reasoning (Zelkova) to validate policies against your security standards.
  • CheckNoNewAccess — verifies a policy update doesn't grant new permissions compared to a reference policy
  • CheckAccessNotGranted — verifies a policy does not grant access to specified actions or resources
  • CheckNoPublicAccess — verifies a resource policy does not grant public access

Other IAM Updates

  • Policy Validation: Over 100 policy checks for syntax, security warnings, errors, and suggestions
  • Policy Generation: Generates least-privilege policies from CloudTrail activity (up to 90 days of history)
  • ABAC (Attribute-Based Access Control): Tag-based access control using aws:PrincipalTag, aws:ResourceTag, and aws:RequestTag conditions. Reduces policy count by using tags instead of per-resource ARN policies.
EXAM TIP: IAM Roles Anywhere is the answer for hybrid scenarios where on-prem workloads need AWS access without long-term keys. Know all three Access Analyzer types and that unused access analyzers are NOT regional. Custom Policy Checks are paid and integrate into CI/CD for policy validation.

5. AWS Security Hub

Updated SA Pro Security
Branding update: AWS Security Hub is now positioned as "Security Hub CSPM" (Cloud Security Posture Management), reflecting its role as the central posture management service.

Cross-Region Aggregation

  • Aggregate findings from multiple Regions to a single home Region
  • Bidirectional replication: findings and status updates flow both ways
  • Simplifies compliance dashboards and centralized monitoring

Central Configuration (Major Update)

The delegated administrator can now configure Security Hub across all accounts and Regions from a single home Region.

Feature Description
Configuration Policies Define which security standards and controls are enabled. Up to 20 custom policies. Attached to OUs or accounts.
Centrally Managed Account/OU is governed by the delegated admin's configuration policy. Local admins cannot override.
Self-Managed Account/OU manages its own Security Hub configuration. Delegated admin designates it as self-managed.
Drift Prevention Centrally managed accounts automatically revert if someone manually changes configuration.

Other Enhancements

  • ASFF (AWS Security Finding Format): Standardized JSON format for all findings — used by GuardDuty, Inspector, Macie, Firewall Manager, and third-party integrations
  • Consolidated control findings: Single control IDs across security standards (e.g., EC2.19 instead of duplicate findings from CIS and PCI DSS)
  • Security scores: Overall and per-standard scores based on percentage of passed controls vs total enabled controls
  • Automations: Rules to automatically update, suppress, or route findings based on criteria
EXAM TIP: Central Configuration is exam-critical. Understand the difference between centrally managed (delegated admin controls everything, drift prevention) vs self-managed (account controls its own settings). Know that Security Hub uses ASFF format, while Security Lake uses OCSF format.

6. Amazon GuardDuty

Updated SA Pro Security
API change (March 2023): The dataSources parameter was replaced by features in the API. GuardDuty capabilities are now referred to as protection plans.

Protection Plans

Protection Plan Data Source What It Detects Notes
Foundational CloudTrail management events, VPC Flow Logs, DNS query logs Account compromise, unauthorized access, reconnaissance Always on when GuardDuty is enabled. Uses independent log streams (no cost to you for the logs).
S3 Protection CloudTrail S3 data events Suspicious S3 access patterns, data exfiltration Monitors object-level API operations
EKS Protection EKS audit logs Suspicious Kubernetes activity, compromised pods Control plane audit log analysis
Lambda Protection Lambda network activity (VPC flow logs) Compromised Lambda functions, crypto mining, C&C communication Monitors Lambda function networking behavior
RDS Protection RDS login activity Brute force attacks, suspicious logins to Aurora and RDS ML-based anomaly detection on database login behavior
Malware Protection for EC2 EBS volume snapshots Malware, trojans, crypto miners on EC2 instances Creates snapshot of EBS volumes and scans in an isolated account. Does not impact running instances.
Malware Protection for S3 Newly uploaded S3 objects Malware in uploaded files Can run independently without GuardDuty being enabled
Runtime Monitoring OS-level events (system calls, file access, network, process) Runtime threats on containers and instances: privilege escalation, reverse shells, file tampering Covers EKS, ECS (Fargate), and EC2. 30+ finding types. Uses a security agent.
AWS Backup Scanning Backup vault recovery points Malware in backup data Scans recovery points before restore
Malware Protection for S3 independence: Unlike all other protection plans, Malware Protection for S3 can operate independently without enabling the core GuardDuty service. This is significant for architectures that only need S3 object scanning.

Extended Threat Detection

  • Detects multi-stage attack sequences across multiple data sources and time periods
  • Correlates individual findings into comprehensive attack narratives
  • Automatically enabled for all GuardDuty accounts — no additional cost
  • Generates new finding type: AttackSequence with critical severity
EXAM TIP: Know ALL protection plans and their data sources. Key facts: (1) Malware Protection for S3 can run independently. (2) Runtime Monitoring covers EKS + ECS Fargate + EC2. (3) Foundational sources (CloudTrail, VPC Flow Logs, DNS) are always on. (4) RDS Protection uses ML for login anomaly detection. (5) GuardDuty detects threats — it does NOT remediate them.

7. AWS Verified Access

NEW SA Pro Security Networking

GA April 2023. Provides zero-trust network access to corporate applications without requiring a VPN.

How It Works

  • Evaluates every request against identity and device posture policies before granting access
  • No VPN client required — users access applications directly through endpoints
  • Per-application access control rather than broad network-level access

Key Components

Component Description
Verified Access Instance The top-level container that holds trust providers and groups
Trust Providers Identity providers (IAM Identity Center, third-party OIDC IdPs) and device management providers (CrowdStrike, Jamf, etc.)
Access Groups Collections of endpoints with shared policies
Access Policies Cedar language policies evaluated on each request
Endpoints Application entry points (ALB, network interface, or ENI)

Supported Protocols

  • HTTP / HTTPS (original support)
  • SSH, TCP, RDP (added for non-web workloads)

Integrations

  • WAF: Attach AWS WAF web ACLs for additional filtering
  • Signed identity context: Passes verified user identity to backend applications via signed HTTP headers
  • Logging: Access logs to CloudWatch, S3, or Kinesis Data Firehose
EXAM TIP: When you see "replace VPN with zero trust" or "application-level access without VPN" in an exam question, the answer is AWS Verified Access. It evaluates identity + device posture per request, unlike VPNs which grant broad network access.
AWS Docs: AWS Verified Access User Guide

8. Amazon Security Lake

NEW Security

GA May 2023. Centralized security data lake that aggregates, normalizes, and stores security data in your own S3 buckets.

Core Concepts

  • OCSF (Open Cybersecurity Schema Framework): All data is normalized to this open standard schema, enabling consistent querying across sources
  • Apache Parquet + Apache Iceberg: Data is stored in Parquet format using Iceberg tables for efficient querying
  • Your S3 buckets: Data stays in S3 buckets in YOUR account — you own the data and control access

Data Sources

  • AWS CloudTrail management and data events
  • Amazon VPC Flow Logs
  • Amazon Route 53 Resolver query logs
  • AWS Security Hub findings
  • Amazon EKS audit logs
  • AWS WAF logs
  • 50+ third-party integrations (Splunk, CrowdStrike, Palo Alto, etc.)
  • Custom sources via OCSF-formatted data

Key Features

  • Multi-account / Multi-region: Centralize via AWS Organizations with delegated admin
  • Subscriber access: Grant query access or data access to security tools (e.g., Splunk, Athena)
  • Detective integration: Amazon Detective can query Security Lake raw logs for deeper investigation
  • Lifecycle management: Automatic rollup regions and retention configuration
Security Lake vs Security Hub:
Security Lake = raw log data in OCSF format stored in S3. Think "data lake for security logs."
Security Hub = aggregated findings in ASFF format. Think "findings dashboard and posture management."
They are complementary — Security Hub findings are one of the sources that feed into Security Lake.
EXAM TIP: Know that Security Lake uses OCSF while Security Hub uses ASFF. Security Lake stores raw logs; Security Hub stores processed findings. If the question asks about a centralized security data lake with raw log analysis, the answer is Security Lake.
AWS Docs: Amazon Security Lake User Guide

9. CloudTrail Lake

Updated Sunsetting Security

Managed data lake for SQL-based querying of CloudTrail audit events, configuration items, and non-AWS event sources.

Key Features

  • Event data stores: Immutable, append-only storage with configurable retention (up to 3,653 days / ~10 years)
  • SQL querying: Use standard SQL to query events directly without needing Athena or Glue
  • Multi-source ingestion: AWS events (management, data, Insights, Config items) plus non-AWS events via integrations
  • Organization-wide: Aggregate events from all accounts in an Organization into a single event data store
  • Dashboards: Built-in visualization of query results
SUNSETTING: CloudTrail Lake ends support for new customers on May 31, 2026. Existing customers can continue use, but AWS is directing customers toward CloudWatch-based capabilities for log querying and analysis. Plan migration accordingly.
EXAM TIP: CloudTrail Lake may still appear on exams taken before the sunset. Know that it provides SQL-based querying of audit events with immutable, configurable retention. For new architectures, consider CloudWatch or Security Lake alternatives.
AWS Docs: CloudTrail Lake

10. AWS KMS Changes

Updated SA Pro Security

External Key Stores (XKS) — November 2022

Store and use encryption keys outside of AWS, in your own external key manager or HSM.

Feature Detail
How it works XKS proxy (on-prem or in your data center) mediates between KMS and your external key manager
Encryption model Double encryption: data is encrypted by KMS AND by the external key manager
Key types supported Symmetric encryption only (AES-256)
Rotation Automatic rotation NOT supported
Multi-Region NOT supported — XKS keys are single-Region
Use case Regulatory requirements mandating keys remain outside AWS (sovereignty, compliance)

Multi-Region Keys — June 2021

Feature Detail
How it works Same key ID and key material replicated across multiple Regions
Architecture One primary key + one or more replica keys in other Regions
Key types Symmetric, asymmetric, and HMAC keys
Use case Cross-region encrypt/decrypt without re-encryption, cross-region DynamoDB global tables, disaster recovery

Other KMS Updates

  • HMAC keys: Generate and verify HMAC tags using KMS-managed keys (HMAC_224, HMAC_256, HMAC_384, HMAC_512)
  • FIPS 140-3 Level 3 validation: Upgraded from FIPS 140-2 Level 2 (standard) / Level 3 (CloudHSM-backed), now all KMS HSMs are FIPS 140-3 Level 3 validated
EXAM TIP: XKS = "keys must remain outside AWS" (sovereignty/regulatory). Multi-Region keys = cross-region encrypt/decrypt without re-encryption. Remember XKS limitations: symmetric only, no rotation, no multi-region. Multi-Region keys support all key types.

11. AWS Private CA

Updated Security
Rename: ACM Private CA was renamed to AWS Private CA — it is now a standalone service, separate from AWS Certificate Manager (ACM).

Two Operating Modes

Feature General-Purpose Mode Short-Lived Mode
Certificate validity Any validity period Maximum 7 days
Revocation (CRL/OCSP) Supported Not supported (certificates expire too quickly to need revocation)
Cost Higher per-CA monthly fee Lower cost per-CA monthly fee
Use cases TLS, code signing, enterprise PKI IoT devices, ephemeral workloads, containers, microservices

Key Features

  • Post-quantum cryptography: Supports ML-DSA (Module-Lattice-Based Digital Signature Algorithm) for quantum-resistant certificates
  • IAM Roles Anywhere integration: Acts as the Trust Anchor for IAM Roles Anywhere, enabling external workloads to authenticate with X.509 certificates issued by your Private CA
  • Cross-account sharing: Share CAs across accounts using AWS RAM
EXAM TIP: Short-lived mode is the answer for IoT and ephemeral workloads where certificates rotate frequently and revocation overhead is unnecessary. Remember the integration path: Private CA issues certificates → IAM Roles Anywhere uses them as trust anchors → external workloads get temporary AWS credentials.

Security
  • Finding groups: Automatically groups related findings into a single incident view, showing related entities and timelines
  • Automated investigation: One-click investigation that analyzes IAM entities and generates a summary of anomalous behavior

Amazon Macie

Security
  • Automated sensitive data discovery for S3 only
  • Uses ML and pattern matching to identify PII, financial data, credentials, and custom data types

19. Cross-Service Exam Themes

SA Pro DevOps Security

Delegated Administrator Pattern

Best practice: Designate a dedicated security tooling account (not the management account) as the delegated administrator for security services. The management account should have minimal workloads and tightly restricted access. Most security services now support delegated admin: Security Hub, GuardDuty, Detective, Macie, Inspector, Firewall Manager, IAM Access Analyzer, Config aggregator, and more.

Multi-Account Governance Stack

The recommended layered approach for multi-account security governance:

Layer Service Role
1. Account Structure AWS Organizations OU hierarchy, SCPs, RCPs, Declarative Policies
2. Account Governance AWS Control Tower Landing zone, preventive/detective/proactive controls, account factory
3. Identity IAM Identity Center Centralized SSO, permission sets, identity federation
4. Posture Management Security Hub Aggregated findings, compliance scoring, central configuration
5. Threat Detection GuardDuty Continuous threat monitoring across all protection plans
6. Investigation Detective Root cause analysis of security findings

Three Policy Types at a Glance

Policy Type Governs Mechanism Exam Keyword
SCPs Principals (IAM users/roles) IAM policy evaluation "Restrict what users/roles can do"
RCPs Resources (S3, SQS, KMS, etc.) Resource policy evaluation "Restrict who can access resources"
Declarative Policies Service configuration Service control plane "Enforce service settings org-wide"

Detection vs Investigation vs Posture

Service Role Scope Key Distinction
GuardDuty Threat detection Multi-service (CloudTrail, VPC, DNS, S3, EKS, Lambda, RDS, EC2, ECS) Finds threats; does not remediate or investigate
Detective Investigation Ingests GuardDuty, CloudTrail, VPC Flow, EKS, Security Lake Root cause analysis; answers "what happened and why"
Security Hub Posture management Aggregates findings from all security services Compliance scoring; centralized findings; ASFF format
Macie Data classification S3 only Finds sensitive data; S3-specific
Inspector Vulnerability management EC2, ECR images, Lambda functions Software vulnerabilities and unintended network exposure

Zero Trust Architecture

AWS Verified Access is the primary service for zero-trust network access. It replaces traditional VPN-based access with per-request evaluation of identity and device posture. Combine with: IAM Identity Center (identity), third-party device trust providers (device posture), WAF (request filtering), and CloudWatch (logging).

Hybrid Key Management & Identity

The hybrid trifecta:
IAM Roles Anywhere — temporary AWS credentials for on-prem workloads via X.509 certificates
KMS External Key Stores (XKS) — encryption keys stored outside AWS for sovereignty requirements
AWS Private CA — managed certificate authority that issues certificates for Roles Anywhere and internal PKI
Together these enable secure hybrid operations with no long-term AWS access keys and full key sovereignty.
